title: DAG Workflows and Jobs
description: Execute multi-step workflows with task dependencies and jobs with execution modes

This example demonstrates two orchestration methods in Propeller:

1. **Workflows** — Define explicit task dependencies using a DAG structure. Tasks specify which other tasks they depend on via `depends_on`.
2. **Jobs** — Group tasks with a shared execution mode (`parallel` or `sequential`). No explicit dependencies—the mode controls execution order.

Both use the same underlying tasks but differ in how execution order is determined.

---

## Part 1: DAG Workflow

This section demonstrates [DAG (Directed Acyclic Graph) workflow](/task-scheduling#workflows) execution. Workflows allow you to define task dependencies where child tasks only execute after their parent tasks complete successfully.

### Prerequisites

Ensure you have:

- Propeller services running (`make start-all`)
- A [provisioned config](/getting-started#step-3-provision-propeller) with valid credentials
- The `addition.wasm` module built

### Source Code

This example uses the [addition-wat](https://github.com/absmach/propeller/tree/main/examples/addition-wat) example, which exports a `main` function that adds two i32 integers:

```lisp
(module
    (func (export "main") (param i32 i32) (result i32)
        local.get 0
        local.get 1
        i32.add))
```

### Build WASM

Propeller executes tasks as WebAssembly modules, so we need to compile the source code to `.wasm` format before uploading.

Build the addition module from WAT (WebAssembly Text format):

```bash
cd propeller
wat2wasm examples/addition-wat/addition.wat -o build/addition.wasm
```

The command produces no output on success.

Verify the build:

```bash
file build/addition.wasm
```

Your output should look like this:

```text
build/addition.wasm: WebAssembly (wasm) binary module version 0x1 (MVP)
```

This confirms the file is a valid WebAssembly binary. "Version 0x1 (MVP)" indicates WebAssembly 1.0, the first standardized version supported by all runtimes.

### Create Workflow

Create a DAG workflow with three tasks. Three tasks is the minimum needed to demonstrate [parallel fan-out](/task-scheduling#fan-out-and-fan-in): Task 1 is the root, and Task 2 and Task 3 both depend on it. When Task 1 completes, Tasks 2 and 3 start simultaneously.

![Example DAG](../images/dag/example-dag.svg)

Task IDs are auto-generated by Propeller if omitted. In this example, we explicitly set `"id": "task-001"` so that `task-002` and `task-003` can reference it in their `depends_on` arrays:

```bash
curl -X POST "http://localhost:7070/workflows" \
-H "Content-Type: application/json" \
-d '{
  "tasks": [
    {
      "id": "task-001",
      "name": "main",
      "inputs": [10, 20]
    },
    {
      "id": "task-002",
      "name": "main",
      "inputs": [5, 15],
      "depends_on": ["task-001"]
    },
    {
      "id": "task-003",
      "name": "main",
      "inputs": [100, 200],
      "depends_on": ["task-001"]
    }
  ]
}'
```

### Upload WASM

Since we built `addition.wasm` locally, we upload it directly to each task. For production workflows, you can use `image_url` instead—see [WASM Module Deployment](/task-scheduling#wasm-module-deployment).

```bash

```

### Start Workflow

Start the root task (`task-001`). The manager automatically schedules dependent tasks (`task-002` and `task-003`) when their dependencies complete:

```bash
curl -X POST "http://localhost:7070/tasks/task-001/start"
```

Your output should look like this:

```json
{ "started": true }
```

## Part 2: Job

This section demonstrates [job-based execution](/task-scheduling#jobs). Jobs group tasks with a shared execution mode—no explicit dependencies. The mode controls whether tasks run in parallel or sequentially.

### Create Job

Create a job with `parallel` execution mode. All tasks in a parallel job start simultaneously:

```bash
curl -X POST "http://localhost:7070/jobs" \
-H "Content-Type: application/json" \
-d '{
  "name": "addition-pipeline",
  "execution_mode": "parallel",
  "tasks": [
    {
      "id": "job-task-001",
      "name": "main",
      "inputs": [100, 50]
    },
    {
      "id": "job-task-002",
      "name": "main",
      "inputs": [25, 75]
    },
    {
      "id": "job-task-003",
      "name": "main",
      "inputs": [200, 300]
    }
  ]
}'
```

Key differences from workflows:

### Sequential Job

Create a job with `sequential` execution mode. Use `depends_on` chains so each task starts only after the previous one completes:

```bash
curl -X POST "http://localhost:7070/jobs" \
-H "Content-Type: application/json" \
-d '{
  "name": "sequential-pipeline",
  "execution_mode": "sequential",
  "tasks": [
    {
      "id": "seq-task-001",
      "name": "main",
      "inputs": [1, 2]
    },
    {
      "id": "seq-task-002",
      "name": "main",
      "inputs": [3, 4],
      "depends_on": ["seq-task-001"]
    },
    {
      "id": "seq-task-003",
      "name": "main",
      "inputs": [5, 6],
      "depends_on": ["seq-task-002"]
    }
  ]
}'
```

Upload WASM and start the job:

```bash
curl -X PUT "http://localhost:7070/tasks/seq-task-001/upload" -F "file=@build/addition.wasm"
curl -X PUT "http://localhost:7070/tasks/seq-task-002/upload" -F "file=@build/addition.wasm"
curl -X PUT "http://localhost:7070/tasks/seq-task-003/upload" -F "file=@build/addition.wasm"
```

### Execution Mode Comparison

| Mode         | Behavior                                                                              | Start Times (this example)               |
| ------------ | ------------------------------------------------------------------------------------- | ---------------------------------------- |
| `parallel`   | All root tasks start simultaneously                                                   | `.452`, `.468`, `.485` (33ms spread)     |
| `sequential` | Root task starts first; each `depends_on` task starts after its predecessor completes | `.099`, `.250`, `.391` (non-overlapping) |

The `depends_on` chains enforce the ordering — seq-task-002 starts only after seq-task-001 finishes, and seq-task-003 starts only after seq-task-002 finishes. If any task fails, the job stops immediately (fail-fast).

For a comparison of workflows vs jobs and detailed job configuration options, see [Workflow vs Job Comparison](/task-scheduling#workflow-vs-job-comparison).
